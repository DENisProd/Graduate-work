# ТЗ: Логика приглашений пользователей (фронтенд)

Документ описывает сценарии использования приглашений в дом, порядок вызовов API и передаваемые данные. Базовый URL API: `{BASE_URL}/api/v1`. Для авторизованных запросов передаётся заголовок **`X-User-Id`** — внешний (системный) ID текущего пользователя.

---

## 1. Обзор API приглашений

| Метод | URL | Описание |
|-------|-----|----------|
| POST | `houses/:houseId/invitations` | Создать приглашение в дом |
| POST | `invitations/accept` | Принять приглашение (по токену) |
| DELETE | `invitations/:id` | Отозвать приглашение |

**Статусы приглашения** (enum `InvitationStatus`): `PENDING`, `ACCEPTED`, `DECLINED`, `REVOKED`, `EXPIRED`.

---

## 2. Сценарии использования

### 2.1. Владелец/админ отправляет приглашение в дом

**Кто:** Пользователь, имеющий право приглашать в дом (владелец или участник с соответствующей ролью).

**Шаги:**

1. Пользователь открывает экран настроек дома или «Участники» и нажимает «Пригласить».
2. Вводит **email** приглашаемого и при необходимости выбирает **роль** и **срок действия** приглашения.
3. Фронтенд отправляет запрос создания приглашения (см. п. 3.1).
4. Бэкенд возвращает созданное приглашение и **токен** (возвращается только при создании).
5. Фронтенд формирует ссылку для приглашённого, подставляя токен в URL, например:  
   `https://<frontend-host>/invite?token=<token>`  
   Токен нужно сохранить только в этой ссылке; для последующего отзыва приглашения используется **id** приглашения.
6. Отобразить пользователю успех и, при необходимости, скопировать ссылку или отправить её по email (отправка письма — отдельная задача/интеграция).

**Важно:** Токен в ответе создания приглашения приходит **один раз**. Для отображения списка отправленных приглашений и отзыва понадобится либо хранение созданных приглашений на фронте (id, email, status, expiresAt), либо эндпоинт вида `GET houses/:houseId/invitations` (на момент написания ТЗ в API не описан — при появлении использовать его для списка и отзыва по `id`).

---

### 2.2. Пользователь принимает приглашение

**Кто:** Приглашённый пользователь (тот, кому отправили ссылку с токеном).

**Предусловие:** Пользователь переходит по ссылке с токеном (например `/invite?token=...`) и при этом **уже авторизован** в системе (его `userId` известен фронту).

**Шаги:**

1. Фронтенд извлекает `token` из query-параметра URL (`?token=...`).
2. Если пользователь не авторизован — показать экран входа/регистрации, после входа вернуться к экрану приглашения с тем же `token`.
3. Показать экран «Принять приглашение» (можно отобразить название дома, если есть данные).
4. При нажатии «Принять» отправить запрос принятия приглашения (см. п. 3.2) с текущим `userId` и этим `token`.
5. При успешном ответе показать успех и перенаправить в дом (например на страницу дома или список домов).
6. Обработать ошибки: «Приглашение уже обработано», «Приглашение истекло», «Приглашение не найдено» — показать соответствующие сообщения.

**Альтернатива (без авторизации до перехода по ссылке):** после входа/регистрации фронт сохраняет `token` (в URL или sessionStorage), затем на экране «Мои приглашения» или сразу после входа предлагает принять приглашение — вызов того же `POST invitations/accept` с сохранённым токеном и текущим `userId`.

---

### 2.3. Владелец/админ отзывает приглашение

**Кто:** Пользователь, имеющий право управлять приглашениями дома.

**Шаги:**

1. На экране «Участники» или «Приглашения» дома отображается список отправленных приглашений (из локального состояния или из `GET houses/:houseId/invitations`, когда эндпоинт будет реализован).
2. Для приглашения со статусом `PENDING` доступна кнопка «Отозвать».
3. По нажатию отправить запрос отзыва (см. п. 3.3), передавая **id** приглашения и заголовок `X-User-Id`.
4. После успешного ответа обновить список (удалить или пометить приглашение как отозванное).

**Ограничение бэкенда:** отозвать можно только приглашения в статусе `PENDING`.

---

## 3. Вызовы API: данные и порядок

### 3.1. Создать приглашение

- **Метод и URL:** `POST /api/v1/houses/:houseId/invitations`
- **Заголовки:** `Content-Type: application/json`, `X-User-Id: <externalUserId>` (обязателен).
- **Параметры пути:** `houseId` — UUID дома.
- **Тело запроса (JSON):**

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| email | string | да | Email приглашаемого (валидный email). |
| roleId | string (UUID) | нет | Роль, которая будет назначена при принятии приглашения (ID роли дома). |
| expiresAt | string (ISO 8601) | нет | Дата истечения приглашения. По умолчанию бэкенд ставит 7 дней. |

**Пример тела:**

```json
{
  "email": "user@example.com",
  "roleId": "123e4567-e89b-12d3-a456-426614174000",
  "expiresAt": "2025-03-14T12:00:00.000Z"
}
```

**Ответ (201):** объект приглашения + одноразовый токен:

```json
{
  "id": "uuid-invitation-id",
  "houseId": "uuid-house-id",
  "email": "user@example.com",
  "status": "PENDING",
  "createdAt": "2025-03-07T12:00:00.000Z",
  "acceptedAt": null,
  "expiresAt": "2025-03-14T12:00:00.000Z",
  "invitedById": "uuid-house-member-id",
  "token": "one-time-token-string"
}
```

- **token** — использовать только для формирования ссылки приглашения; в последующих запросах для отзыва используется **id**.
- Сохранить на фронте (при отсутствии GET-списка): `id`, `houseId`, `email`, `status`, `expiresAt` для отображения и отзыва.

---

### 3.2. Принять приглашение

- **Метод и URL:** `POST /api/v1/invitations/accept`
- **Заголовки:** `Content-Type: application/json`. Заголовок `X-User-Id` при вызове accept не обязателен для бэкенда, но **userId** в теле должен соответствовать авторизованному пользователю; при необходимости фронт может передавать `X-User-Id` для единообразия.
- **Тело запроса (JSON):**

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| token | string | да | Токен приглашения (из ссылки или сохранённый после перехода по ссылке). |
| userId | string | да | Внешний (системный) ID текущего пользователя — тот, кто принимает приглашение. |

**Пример тела:**

```json
{
  "token": "one-time-token-string",
  "userId": "external-user-id-from-auth"
}
```

**Ответ (200):** объект приглашения (без поля `token`), статус будет `ACCEPTED`:

```json
{
  "id": "uuid-invitation-id",
  "houseId": "uuid-house-id",
  "email": "user@example.com",
  "status": "ACCEPTED",
  "createdAt": "2025-03-07T12:00:00.000Z",
  "acceptedAt": "2025-03-07T12:05:00.000Z",
  "expiresAt": "2025-03-14T12:00:00.000Z",
  "invitedById": "uuid-house-member-id"
}
```

**Типичные ошибки:**

- 400 «Приглашение уже обработано» — статус не PENDING.
- 400 «Приглашение истекло» — `expiresAt` в прошлом.
- 404 «Приглашение не найдено» — неверный или использованный токен.

---

### 3.3. Отозвать приглашение

- **Метод и URL:** `DELETE /api/v1/invitations/:id`
- **Заголовки:** `X-User-Id: <externalUserId>` (обязателен — от имени кого отзываем).
- **Параметры пути:** `id` — UUID приглашения (из ответа создания или из списка приглашений дома).

**Ответ:** 204 No Content при успехе.

**Ошибки:**

- 400 «Можно отозвать только ожидающие приглашения» — статус не PENDING.
- 404 — приглашение с таким `id` не найдено.

---

## 4. Порядок вызовов по сценариям

**Отправка приглашения (владелец/админ):**

1. `POST houses/:houseId/invitations` с телом `{ email, roleId?, expiresAt? }`, заголовок `X-User-Id`.
2. Сохранить из ответа `id`, `token`; сформировать ссылку `.../invite?token=<token>`; при необходимости сохранить приглашение в локальный список для отображения и отзыва.

**Принятие приглашения (приглашённый):**

1. Получить `token` из URL (или из сохранённого состояния после входа).
2. Убедиться, что пользователь авторизован и известен `userId`.
3. `POST invitations/accept` с телом `{ token, userId }`.
4. По успеху — переход в дом или список домов.

**Отзыв приглашения (владелец/админ):**

1. Иметь `id` приглашения (из ответа создания или из списка).
2. `DELETE invitations/:id` с заголовком `X-User-Id`.
3. Обновить локальный список приглашений (удалить или обновить статус).

---

## 5. Дополнительно для фронтенда

- **Роли дома:** список ролей для выбора при создании приглашения можно получать через существующий эндпоинт ролей дома, например `GET /api/v1/houses/:houseId/roles` (если реализован в вашем API).
- **Список приглашений дома:** при появлении эндпоинта `GET houses/:houseId/invitations` использовать его для отображения списка и получения актуальных `id` и статусов для отзыва; формат ответа согласовать с бэкендом (массив объектов вида `InvitationResponseDto` без поля `token`).
- **Отклонение приглашения:** в текущем API модуля приглашений (modules/invitations) отдельного эндпоинта «отклонить» нет; при необходимости можно трактовать «отклонение» как не вызывать accept и при истечении срока приглашение станет EXPIRED, либо запросить на бэкенде эндпоинт типа `POST invitations/:id/decline`.
