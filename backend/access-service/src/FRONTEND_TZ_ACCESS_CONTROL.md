# ТЗ для фронтенда: панель управления доступом

Документ описывает требования к frontend-части для работы с системой контроля прав из `ACCESS_CONTROL_USAGE.md`.

## 1. Цель

Сделать UI для:

- управления домами, участниками, ролями, ресурсами;
- назначения прав (`RBAC`) и политик (`ABAC`);
- проверки доступа и диагностики причин отказа;
- отображения доступной структуры ресурсов для пользователя.

## 2. Технические условия интеграции

- Базовый префикс API: `/api/v1`.
- Swagger: `/api/docs`.
- Для операций "от имени пользователя" отправлять заголовок `X-User-Id`.
- Формат дат: ISO строка.
- Ключевые enum:
  - `AccessRightType`: `ALLOW | DENY | READ | WRITE`;
  - `PolicySubjectType`: `USER | ROLE | MEMBER | ANYONE`;
  - действия проверки: `READ | WRITE` (в evaluator), `read | write` (в access-control check).

## 3. Роли в интерфейсе

Минимально поддержать 2 режима UI:

- **Администратор дома**: полный доступ к разделам управления.
- **Обычный пользователь**: просмотр доступной структуры и проверка текущего доступа (без админских действий).

> Факт "кто админ" определяется внешней бизнес-логикой приложения/gateway. На стороне этого сервиса управление через `X-User-Id` и проверки на backend.

## 4. Карта экранов

## 4.1 Страница "Мои дома"

Назначение: список домов пользователя.

- **Route**: `/houses`
- **API**: `GET /houses/user/:userId`
- **Функции**:
  - загрузка списка домов;
  - переход в карточку выбранного дома;
  - создание дома (кнопка "Создать дом", если разрешено продуктом).
- **Пустое состояние**: "У вас пока нет домов".

## 4.2 Страница "Карточка дома"

Назначение: хаб для управления домом.
Вкладки "Участники", "Ресурсы" (Роли, ресурсы, политики, права)

- **Route**: `/houses/:houseId`
- **API**:
  - `GET /houses/:id`
  - `GET /house-members/house/:houseId`
  - `GET /house-roles/house/:houseId`
  - `GET /houses/:houseId/resources/tree`
- **Блоки на странице**:
  - информация о доме;
  - краткая статистика: участники, роли, ресурсы;
  - быстрые действия: "Добавить участника", "Создать роль", "Назначить право", "Проверить доступ".

## 4.3 Страница "Участники дома"

Назначение: управление участниками.

- **Route**: `/houses/:houseId/members`
- **API**:
  - `GET /house-members/house/:houseId`
  - `POST /house-members?houseId=...&userId=...`
  - `DELETE /house-members?houseId=...&userId=...`
- **Функции**:
  - список участников;
  - добавление участника по `userId` (внешний UUID);
  - удаление участника с подтверждением;
  - переход в профиль участника.
- **Валидация**:
  - `userId` обязателен;
  - формат UUID.
  Далее блок активных приглашений и кнопка "Показать все" /api/v1/house-invitations/

## 4.4 Страница "Роли дома"

Назначение: CRUD ролей и назначения ролей участникам.

- **Route**: `/houses/:houseId/roles`
- **API**:
  - `GET /house-roles/house/:houseId`
  - `POST /house-roles/house/:houseId`
  - `DELETE /house-roles/:roleId`
  - `POST /house-roles/members/:memberId/roles/:roleId`
  - `DELETE /house-roles/members/:memberId/roles/:roleId`
- **Функции**:
  - список ролей (имя, приоритет, системная/кастомная);
  - создание кастомной роли (`name`, `priority`);
  - удаление роли (для системной роли кнопка disabled);
  - назначение/снятие роли участнику.
- **Валидация**:
  - `name` не пустой;
  - `priority` целое число >= 0.

## 4.5 Страница "Дерево ресурсов"

Назначение: визуализация структуры `дом -> комнаты -> устройства -> функции`.

- **Route**: `/houses/:houseId/resources`
- **API**:
  - `GET /houses/:houseId/resources/tree`
  - `POST /resources`
  - `GET /resources/:id`
- **Функции**:
  - древовидный просмотр с раскрытием узлов;
  - фильтр по типу ресурса;
  - добавление ресурса (modal/side panel):
    - `type`, `houseId`, `parentId`, `externalId`, `name` (если применимо);
  - просмотр карточки ресурса.

## 4.6 Страница "Права доступа (RBAC)"

Назначение: управление и аудит назначенных прав.

- **Route**: `/houses/:houseId/rights`
- **API**:
  - `GET /access-control/rights/house/:houseId?page=&size=&sort=`
  - `POST /access-control/rights`
  - `PUT /access-control/rights/:id`
  - `DELETE /access-control/rights/:id`
  - `GET /access-control/rights/member/:memberId?page=&size=&sort=`
- **Функции**:
  - таблица прав с пагинацией;
  - фильтры:
    - по субъекту (участник/роль),
    - по типу права (`ALLOW`, `DENY`, `READ`, `WRITE`),
    - по активности (`isExpired`, `isActive`);
  - создание права:
    - выбрать `resourceId`;
    - выбрать **либо** `houseMemberId`, **либо** `houseRoleId`;
    - выбрать `accessRightType`;
    - опционально `expiresAt`, `parameters`;
  - редактирование и удаление права.
- **Ключевое правило формы**:
  - нельзя одновременно передать `houseMemberId` и `houseRoleId`.

## 4.7 Страница "Политики (ABAC)"

Назначение: гибкие правила поверх RBAC.

- **Route**: `/houses/:houseId/policies`
- **API**:
  - `GET /houses/:houseId/policies`
  - `POST /policies`
  - `DELETE /policies/:id`
- **Функции**:
  - список политик с сортировкой по `priority` (по возрастанию);
  - создание политики:
    - `name`, `effect`, `subjectType`, `subjectId?`, `resourceId`, `priority`, `condition?`;
  - удаление политики.
- **Валидация**:
  - `priority` >= 0;
  - если `subjectType != ANYONE`, поле `subjectId` обязательно.

## 4.8 Страница "Проверка доступа"

Назначение: диагностический инструмент для админов/поддержки.

- **Route**: `/access-check`
- **Вкладка A (Evaluator)**:
  - `POST /access/check`
  - поля: `userId`, `resourceId`, `action (READ|WRITE)`;
  - вывод: `allowed`, `source`, `rightType`.
- **Вкладка B (By device function)**:
  - `POST /access-check`
  - поля: `userId`, `deviceFunctionId`, `action`;
  - вывод: `{ allow, deny }`.
- **Вкладка C (Diagnostic Access Control)**:
  - `POST /access-control/check`
  - поля: `userId`, `resourceId`, `operationType (read|write)`;
  - вывод: `hasAccess`, `effectiveRightType`, `reason`, `applicableRights[]`.

## 4.9 Страница "Структура доступа пользователя"

Назначение: what-user-can-see для UI клиента.

- **Route**: `/users/:userId/access-structure`
- **API**: `GET /access-structure?userId=<externalUserId>`
- **Функции**:
  - отображение отфильтрованного дерева доступных зон;
  - быстрый переход к функции устройства;
  - использование результата для скрытия недоступных элементов UI.

## 4.10 Страница "Сервисные операции"

Назначение: эксплуатационные действия.

- **Route**: `/maintenance/access`
- **API**:
  - `POST /access-control/cleanup/expired`
  - `POST /permissions/rebuild`
- **Функции**:
  - запуск cleanup истекших прав;
  - запуск rebuild кэша;
  - уведомления об успешном завершении/ошибке.

## 5. Общие UX требования

- Все destructive-операции (delete/revoke) через confirm dialog.
- Для больших списков использовать серверную пагинацию.
- Для select-полей (`member`, `role`, `resource`) поддержать поиск.
- В формах прав и политик показывать inline-подсказки:
  - что `DENY` выше по приоритету;
  - что `priority` меньше = важнее.
- Ошибки API показывать в человекочитаемом виде (message + код).

## 6. Состояния и обработка ошибок

Для каждой страницы обязательно:

- `loading` состояние;
- `empty` состояние;
- `error` состояние с retry.

Частые backend-ошибки, которые нужно обработать:

- `400` - валидация DTO (показывать полевую ошибку);
- `403` - недостаточно прав на действие;
- `404` - ресурс/роль/право не найдено;
- `409`/дубликаты (если вернутся) - конфликт данных.

## 7. Требования к клиентскому API-слою

- Единый HTTP-клиент с автоматической установкой `X-User-Id`.
- Типизированные модели ответов для:
  - rights list/page;
  - policy;
  - access check response;
  - access structure tree.
- Нормализация дат в UI-формат.
- Отмена in-flight запросов при смене страницы/фильтров.

## 8. Матрица соответствия сценариям

Фронтенд должен поддержать сценарии из доменного документа:

1. **Гость читает датчик**:
   - создать `READ` право;
   - проверить `READ`/`WRITE` в "Проверке доступа".
2. **Роль "family" управляет светом**:
   - создать роль;
   - назначить роль участникам;
   - выдать `WRITE` на ресурсы света.
3. **Временный доступ курьеру**:
   - создать право с `expiresAt`;
   - проверить, что после времени доступа правило неактивно.
4. **Явный запрет DENY**:
   - создать `DENY`;
   - в диагностике видеть отказ с причиной.
5. **UI по доступной структуре**:
   - загрузить `access-structure`;
   - отрисовать только доступные элементы.

## 9. Нефункциональные требования

- Адаптивность: desktop-first, корректная работа от 1280px и выше.
- Производительность:
  - таблицы на 200+ строк без заметных лагов;
  - debounce поиска в фильтрах (300-500 мс).
- Локализация: RU (минимум все админские тексты).

## 10. Критерии приемки

Считаем задачу выполненной, если:

- реализованы все страницы из раздела 4;
- все CRUD-операции выполняются через актуальные endpoint-ы;
- сценарии из раздела 8 воспроизводимы через UI без ручных API-вызовов;
- на всех страницах есть `loading/empty/error`;
- ключевые формы валидируют ограничения backend до отправки.
